# ExecPlan: Поэтапный перевод render-контекста на Pydantic-модели

Этот ExecPlan является живым документом. Разделы `Progress`, `Surprises & Discoveries`, `Decision Log` и `Outcomes & Retrospective` должны поддерживаться в актуальном состоянии по мере выполнения работы. Документ ведется в соответствии с `PLANS.md`.

## Цель / Общая картина

После этой работы система сохранит совместимость на внешней границе: `CDARenderer.render(...)` по-прежнему будет принимать `dict`, как и раньше, и существующие тесты не должны ломаться. При этом внутри пути рендеринга данные больше не должны попадать в Jinja2-шаблоны как сырые словари. Перед передачей в `template.render(...)` контекст должен быть преобразован в Pydantic-модели, чтобы правило "парсинг и валидация на границе" соблюдалось не только для пользовательского тела документа, но и для служебного контекста (`header`, `patient`, `author`, `organization`, `event`, `encounter`, `document_info`).

Наблюдаемое поведение после завершения: можно запустить текущие тесты генерации и веб-интерфейса, передать в `render()` комбинацию из `dict` и `BaseModel`, и получить тот же XML, но шаблон будет работать только с атрибутным доступом Pydantic-моделей. Это будет видно по тому, что из шаблонов исчезнут вызовы `.get(...)`, а новый внутренний шаг нормализации будет покрыт тестами.

## Surprises & Discoveries

- Наблюдение: пользовательская часть документа уже валидируется через `ConsultationProtocolV7`, но `CDARenderer.render()` разворачивал `BaseModel` обратно в `dict` через `model_dump()`, тем самым теряя типизированное представление перед самым шаблоном. Исправлено.
- Наблюдение: текущий шаблон зависел от словарного API и смешивал атрибутный доступ с `.get(...)`. Исправлено переводом всех макросов и шаблона на атрибуты.
- Наблюдение: макросы в `core` использовались разными тестами с разными структурами данных (dict vs model), что потребовало унификации через обязательную регистрацию моделей.
- Сюрприз: `CD` и `ADDR` типы в `core` были неполными для существующих макросов, пришлось расширить их полями `version`, `type`, `translation`.

## Контекст и ориентация

Этот репозиторий генерирует CDA XML на базе Jinja2. Входные пользовательские данные описаны в Pydantic-моделях, а затем подставляются в XML-шаблон. Сейчас типизирована только часть пользовательского тела документа.

Ключевые файлы:

- `semd_engine/registry/v290_r7/schema.py` содержит модель `ConsultationProtocolV7` и вложенные пользовательские модели (`DiagnosisModel`, `PrescribedMedication`). Эта модель валидирует только часть данных, введенных пользователем.
- `main.py` содержит демонстрационный FastAPI-интерфейс. Он преобразует form-data во вложенный `dict`, валидирует этот `dict` через `ConsultationProtocolV7`, а затем вызывает `renderer.render("main.xml.j2", HEADER_MOCK, model)`.
- `semd_engine/core/renderer.py` содержит `CDARenderer.render`. Сейчас метод принимает произвольный набор словарей и моделей, но перед шаблоном превращает всё в один `dict`.
- `semd_engine/registry/v290_r7/templates/main.xml.j2` это основной XML-шаблон. Он ожидает ключи верхнего уровня `header`, `patient`, `organization`, `author`, `event`, `encounter`, `document_info`, а также поля из `ConsultationProtocolV7`.
- `tests/test_pydantic_flow.py` и `tests/test_v290_r7_generation.py` закрепляют текущий сценарий смешанного вызова `render()`.

Термин "контекст рендеринга" в рамках этого плана означает полный набор объектов, которые передаются в `template.render(...)`. Сейчас этот контекст словарный. Цель плана: сделать его модельным. Термин "нормализация" означает внутренний шаг, который принимает смешанный вход (`dict` и `BaseModel`) и строит из него один типизированный объект, пригодный для шаблона.

## План работы

Сначала нужно описать типизированный контекст `v290_r7` в новом файле `semd_engine/registry/v290_r7/context.py`. На этом этапе не нужно пытаться обобщить все CDA-сущности для всего проекта. Достаточно покрыть то, что реально читает `main.xml.j2`: модель заголовка документа, модели пациента, автора, организации, события, encounter и блока `document_info`. Там, где уже существуют пригодные общие типы (`II`, `CD`, `ADDR`, `PN` из `semd_engine/core/cda_types.py`), их нужно переиспользовать. Там, где в шаблоне сейчас используются поля `version`, отсутствующие в `CD`, допустимо добавить локальные модели-обертки для конкретного контекста `v290_r7`, вместо немедленного расширения общего ядра. Корневой объект должен описывать полный шаблонный контекст, например `ConsultationProtocolV7RenderContext`, и включать как служебный header/context, так и payload из `ConsultationProtocolV7`.

Затем нужно расширить `semd_engine/core/renderer.py`. Публичная сигнатура `render(self, template_name: str, *data_objects: dict[str, Any] | BaseModel) -> str` должна остаться совместимой. Внутри нужно добавить отдельный шаг, который строит единый объект контекста. Минимально безопасный путь для первого этапа: позволить рендереру опционально регистрировать нормализатор для шаблона или для конкретного renderer instance. Для `v290_r7` этот нормализатор должен принимать смешанные входы, аккуратно объединять их на верхнем уровне, валидировать через корневую Pydantic-модель и передавать в Jinja2 уже объектную структуру. Если нормализатор не задан, старое словарное поведение может быть временно оставлено как fallback только для других шаблонов, чтобы не ломать код вне `v290_r7`.

После появления типизированного контекста нужно обновить `semd_engine/registry/v290_r7/templates/main.xml.j2`. Все обращения вида `event.get('conditions')` и `field.get('version')` должны быть заменены на модельные обращения, например `event.conditions` и `event.type.version or '3.50'`. Итоговый шаблон не должен полагаться на словарные методы. Если в шаблоне есть участки, где сейчас подразумевается отсутствие ключа вместо `None`, это нужно выровнять на уровне модели через `Optional`.

После этого нужно обновить тесты. В `tests/test_pydantic_flow.py` и `tests/test_v290_r7_generation.py` следует сохранить текущие `dict`-фикстуры `header_data`, чтобы доказать обратную совместимость. Поверх этого нужно добавить по крайней мере один тест, который проверяет новый шаг нормализации явно: из `dict + ConsultationProtocolV7` должен получаться корневой `BaseModel`, а вложенные объекты `header`, `patient`, `event` и другие должны быть моделями, а не словарями. Если для этого потребуется небольшой вспомогательный публичный или защищенный метод в `CDARenderer`, он должен иметь минимальную поверхность и понятное имя.

Работу нужно вести поэтапно, с запуском тестов после каждого изолированного изменения. Сначала тесты на модели контекста. Потом тесты рендерера. Потом тесты полного XML. Если какой-то шаг ломает шаблон или веб-интерфейс, нужно исправить типы, а не возвращаться к словарной передаче в шаблон.

## Валидация и приемка

Приемка этого плана основана на наблюдаемом поведении.

Во-первых, вызов `renderer.render("main.xml.j2", header_dict, consultation_model)` должен по-прежнему работать без изменения вызывающего кода. Это проверяется автоматическими тестами и ручным сценарием из `main.py`.

Во-вторых, внутри рендеринга `main.xml.j2` должен использовать только модельный доступ к данным. Это проверяется отсутствием `.get(` в шаблоне и тестом на нормализацию контекста.

В-третьих, XML-результат должен остаться совместимым с текущими проверками: в нем должны присутствовать ожидаемые поля, а существующие тесты генерации, веб-интерфейса и валидации должны проходить.

Минимальный обязательный набор команд приемки:

    uv run pytest tests/test_pydantic_flow.py tests/test_v290_r7_generation.py
    uv run pytest tests/test_web_interface.py tests/test_validation.py

Если после миграции добавится отдельный тест файла контекста, его нужно включить в этот список и зафиксировать обновленное ожидаемое число passed.

Для ручной проверки нужно запустить:

    uv run python main.py

Затем открыть `http://localhost:8001`, заполнить форму, нажать кнопку генерации и убедиться, что XML строится без ошибки в HTML-ответе. Этот сценарий доказывает, что путь `form -> nested dict -> ConsultationProtocolV7 -> render(dict + BaseModel)` продолжает работать.

## Артефакты и примечания

Ключевой текущий фрагмент, который нужно заменить по смыслу:

    def render(self, template_name: str, *data_objects: dict[str, Any] | BaseModel) -> str:
        render_data = {}
        for obj in data_objects:
            if isinstance(obj, BaseModel):
                render_data.update(obj.model_dump())
            else:
                render_data.update(obj)
        template = self.env.get_template(template_name)
        return template.render(**render_data)

После реализации здесь должен появиться явный шаг построения типизированного контекста перед `template.render(...)`.

Ключевые места шаблона, требующие миграции на модельный доступ:

    event.type.get('version')
    event.get('conditions')
    event.conditions.get('version')
    document_info.case_type.get('version')
    document_info.patient_condition.get('version')

Примечание ревизии: этот план создан после исследования текущего рендер-пайплайна и фиксирует согласованные решения: контекстные модели сначала делаются локально для `v290_r7`, `dict`-фикстуры в тестах сохраняются, а отсутствие полей моделируется через `Optional[...]`, а не через тихие дефолты бизнес-данных.

## Интерфейсы и зависимости

В конце первого этапа должны существовать следующие сущности или их эквиваленты:

- В `semd_engine/registry/v290_r7/context.py`:

  `class ConsultationProtocolV7RenderContext(BaseModel): ...`

  Эта модель должна содержать поля верхнего уровня, которые читает `main.xml.j2`: `header`, `patient`, `organization`, `author`, `event`, `encounter`, `document_info`, а также поля payload-модели `ConsultationProtocolV7` или вложенную payload-модель, если это упростит сборку контекста.

- В `semd_engine/core/renderer.py`:

  `def render(self, template_name: str, *data_objects: dict[str, Any] | BaseModel) -> str:`

  Сигнатура остается прежней.

  Внутри должен появиться минимальный механизм нормализации, например:

  `def _normalize_render_context(self, template_name: str, data_objects: tuple[dict[str, Any] | BaseModel, ...]) -> BaseModel | dict[str, Any]: ...`

  или другой эквивалентный метод с тем же смыслом.

- В `main.py`:

  Конфигурация `renderer` должна использовать новый нормализатор для `v290_r7`, если он не может быть выбран автоматически по шаблону.

Зависимости должны оставаться прежними: `pydantic`, `jinja2`, `lxml`, `fastapi`, `pytest`, `uv`. Новые библиотеки для этой задачи не нужны.
