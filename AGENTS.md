# Архитектура генератора (Agents & Components)

## Основные компоненты

### 1. Schema Agent (Модели данных)
Использует Pydantic для декларативного описания структуры документа.
- **UI Metadata**: Модели содержат аннотации для фронтенда (типы инпутов, условия видимости).
- **Validation**: Та же модель служит единственным источником правды для проверки входящего JSON.

### 2. Data Fetcher Agent (Интеграция)
Абстрактный слой, отделяющий генератор от физического хранения данных.
- **Interfaces**: Определяют контракты для получения данных пациентов, врачей и справочников НСИ.
- **Isolation**: Позволяет использовать любые источники (SQL, REST API, Redis) без изменения логики генератора.

### 3. Rendering Engine (Генерация XML)
Движок на базе Jinja2, использующий строгую типизацию.
- **CDA Macros**: Библиотека переиспользуемых XML-блоков для соответствия стандарту CDA R2.
- **Validation Suite**: 
    - Встроенная XSD-валидация для проверки структуры.
    - Внешняя Schematron-валидация (через `schxslt-cli.jar`) для проверки бизнес-логики (XSLT 2.0).
- **Strict Mode**: Гарантирует, что ошибки в данных будут выявлены на этапе сборки документа.

## Принципы
- **Metadata-Driven**: Логика UI и валидации управляется метаданными в коде.
- **Loose Coupling**: Генератор ничего не знает о внутреннем устройстве основной системы.
- **Type Safety**: Максимальное использование аннотаций типов для предотвращения ошибок.
- **Traceability**: Актуальный статус реализации и принятые технические решения фиксируются в `PROGRESS.md`.
- **Environment**: Использование `uv` для управления виртуальным окружением и зависимостями.

# ExecPlans

Для каждого типа документа или сложной задачи создается план реализации в директории `execplans/`.
- Планы помогают отслеживать прогресс реализации конкретных СЭМД.
- Включают в себя шаги от анализа схем до финальной валидации.
- Пример: `execplans/v290_r7_consultation.md`.

При написании сложных функций или значительных рефакторингов используйте ExecPlan — от проектирования до реализации.

# СЭМД: Tips & Gotchas

### 1. Строгий порядок в CDA R2 Header
XSD-схема CDA (POCD_MT000040) требует строгого порядка элементов. 
**Правильный порядок:** `realmCode` -> `typeId` -> `templateId` -> `id` -> `code` -> `title` -> `effectiveTime` -> `confidentialityCode` -> `languageCode` -> `setId` -> `versionNumber` -> `recordTarget` -> `author` -> `custodian` -> `informationRecipient` -> `documentationOf` -> `componentOf`.
*Нарушение порядка приведет к ошибке XSD, даже если все элементы на месте.*

### 2. Магия суффиксов OID
В российских СЭМД идентификаторы (`id`, `setId`) часто должны заканчиваться на специфические суффиксы для прохождения Schematron-валидации:
- `.51` — `ClinicalDocument/id` (Номер документа)
- `.50` — `ClinicalDocument/setId` (Номер случая/амбулаторной карты)
- `.52` — `observation/id` (Идентификатор записи, например, диагноза)
- `.15` — `encompassingEncounter/id`

### 3. Особенности вложенности (entryRelationship)
Бизнес-логика в Schematron часто требует глубокой вложенности. Например, диагноз `838` ОБЯЗАН содержать `entryRelationship` с кодом `1` (Тип диагноза), который в свою очередь содержит вложенные `entryRelationship` для МКБ-кода (`12014`) и характера заболевания (`823`).
*Проверяйте правила `У3-38.1.5` в Schematron для понимания иерархии.*

### 4. Валидация и Namespaces
Schematron-валидатор (особенно через `schxslt-cli.jar`) чувствителен к пространствам имен. 
- Сохраняйте российские расширения (`identity`, `address`, `medService`, `fias`), так как они используются в XPATH-выражениях правил.
- `CDARenderer` автоматически обрабатывает дефолтный namespace (`urn:hl7-org:v3`) перед валидацией для корректной работы XSLT-правил.

### 5. Артефакты документации (Reference)
При разработке нового типа документа используйте следующие файлы как эталон (находятся в папке с версией СЭМД):
- `guide.txt` / `Description.pdf`: текстовое описание полей и правил.
- `m0134_sample.xml` (или аналоги): пример валидного XML, на который стоит ориентироваться при построении структуры.
- `.sch` файл: содержит финальную истину о бизнес-правилах (иерархия, коды, условия).

### 6. Формат дат (HL7 TS)
Даты должны быть в формате `YYYYMMDDHHMMSS+ZZZZ`. В шаблонах используйте фильтр `| hl7date` (реализованный в `CDARenderer`), который приводит стандартные Python datetime/ISO-строки к этому формату.
